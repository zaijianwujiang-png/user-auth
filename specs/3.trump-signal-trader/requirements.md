# trump-signal-trader — 需求规格

## 概述

抓取特朗普 Truth Social（@realDonaldTrump）新帖，用 Claude 双模型解析为加密货币交易信号并做多层交叉验证，通过后推送 Telegram。**v1 只做「信号 + 通知」，完全不碰交易下单。**

## 项目信息

- 项目名: trump-signal-trader
- 架构类型: 单体 Serverless（Python Lambda + EventBridge + DynamoDB）
- 用户: 单用户个人量化工具，不对外服务

## 需求版本

| 日期       | 版本 | 说明                          |
| ---------- | ---- | ----------------------------- |
| 2026-07-07 | v1   | 初始需求（采集+信号+验证+通知）|

## 用户故事

- 作为交易者，我想在特朗普发帖后 1 分钟内收到 AI 解析的结构化交易信号，以便快速判断是否手动跟进。
- 作为交易者，我想让重复内容、旧闻、双模型分歧、已被市场消化、疑似假消息的信号被自动否决，以便只收到高置信度提醒。
- 作为运维者（我自己），我想在采集断线/被限流/系统异常时收到告警，以便及时干预。

## 功能需求（v1）

1. [F-001] 系统定时轮询 @realDonaldTrump 的 Truth Social 帖子，发现新帖到产出通知的目标延迟 ≤ 1 分钟（受轮询周期限制，最坏 ~2 分钟可接受）。
2. [F-002] 采集失败（网络错误/被限流/结构变化）时自动切换降级数据源并推送 Telegram 告警；恢复后自动切回。
3. [F-003] 系统用 Claude API 将帖子解析为结构化信号：资产（BTC/ETH/DOGE…）、方向（利多/利空/无关）、置信度（0-1）、理由摘要；「与加密货币无关」的帖子标记后直接跳过后续验证。
4. [F-004] 交叉验证第 1 层——去重与旧闻：同一帖子 ID 只处理一次；内容哈希重复（转发/复读旧消息）不触发信号。
5. [F-005] 交叉验证第 2 层——双模型独立判断：claude-fable-5 与 claude-sonnet-5 各自独立解析，资产与方向一致才通过；分歧则否决并记录双方结论。
6. [F-006] 交叉验证第 3 层——行情验证：拉取币安公开行情（价格/成交量），若目标资产在帖子发布前后已明显异动（price-in）则否决。
7. [F-007] 交叉验证第 4 层——事实核查：联网搜索确认消息真伪（防盗号/假消息），核查不通过则否决。
8. [F-008] 任一验证层否决即终止流程，否决原因落库并可（按配置）推送 Telegram。
9. [F-009] 全部验证通过的信号推送 Telegram：帖子摘要、资产、方向、双模型置信度、行情快照、核查结论。
10. [F-010] 每个环节（新帖发现、信号提取、各层验证结果、系统异常）的完整决策链路写入 DynamoDB，可按帖子 ID 回查。
11. [F-011] 阈值与开关（轮询周期、置信度门槛、price-in 判定参数、是否推送否决消息）通过环境变量/SAM 参数配置，不改代码。

## 非功能需求

- 安全: Anthropic API key、Telegram Bot Token 等密钥一律走环境变量/SAM 参数（NoEcho），不进代码库；日志不打印密钥与完整帖子外的敏感信息。
- 可靠性: 7×24 无人值守；单次轮询失败不影响下一轮；Lambda 异常有告警。
- 成本: LLM 只在「新帖且非重复」时调用；无关帖在第一次提取即短路，避免浪费双模型与搜索调用。
- 合规: 采集只使用公开可访问数据，优先合规稳定方案；请求频率克制（≥30s 间隔）。

## 验收标准（EARS 风格）

- [ ] [AC-001] 当 @realDonaldTrump 发布一条新帖时，系统应在下一个轮询周期内抓到该帖并写入 DynamoDB（状态 `FETCHED`）。
- [ ] [AC-002] 当抓到的帖子 ID 已存在于库中时，系统应跳过且不产生任何 LLM 调用。
- [ ] [AC-003] 当帖子内容哈希与近 30 天内已处理内容相同（转发/复读）时，系统应否决并记录原因 `DUPLICATE`。
- [ ] [AC-004] 当帖子与加密货币无关时，系统应在首次提取后标记 `IRRELEVANT` 并跳过双模型/行情/核查环节。
- [ ] [AC-005] 当双模型对资产或方向判断不一致时，系统应否决并记录双方结论（原因 `MODEL_DISAGREE`）。
- [ ] [AC-006] 当币安行情显示目标资产已明显异动（超过配置阈值）时，系统应否决（原因 `PRICED_IN`）并记录行情快照。
- [ ] [AC-007] 当联网核查无法确认消息真实性时，系统应否决（原因 `FACT_CHECK_FAILED`）。
- [ ] [AC-008] 当信号通过全部验证层时，系统应发送包含资产/方向/置信度/行情快照的 Telegram 消息，且决策记录状态为 `SIGNAL_SENT`。
- [ ] [AC-009] 当主采集源连续失败（≥ 配置次数）时，系统应切换降级源并发送 Telegram 告警。
- [ ] [AC-010] 当给定任意帖子 ID 时，应能从 DynamoDB 查出该帖完整决策链路（各层结果与时间戳）。
- [ ] [AC-011] 代码库中不出现任何真实密钥（grep 校验），密钥均来自环境变量/SAM 参数。

## 依赖

- Claude API（claude-fable-5 / claude-sonnet-5，含 web search 工具）
- 币安公开行情 REST API（无需 API key）
- Telegram Bot API
- Truth Social 公开数据源（主方案 + 降级方案，见 design.md）
- AWS: Lambda / EventBridge / DynamoDB / SAM（沿用现有栈）

## 开放问题

- Truth Social 未认证接口的可用性随平台策略变化，需在实现期实测；若两条方案均不可用，考虑付费第三方聚合（见 design.md 候选）。
- 事实核查的「真伪」判定标准较主观，v1 采用保守策略（查不到佐证即否决），后续按误杀率调参。
